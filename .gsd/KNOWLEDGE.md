# Knowledge Register

<!-- Append-only. Rules, patterns, and lessons learned for future agents. -->

| # | When | Context | Rule / Pattern | Why |
|---|------|---------|----------------|-----|
| K001 | M005-8pv12q planning | Evidence discipline design | Training data recall is never "observed" — it is a source of concepts and hypotheses, not facts. Version numbers, API signatures, CLI flags, config schemas from training data require verification before acting on them. | User constraint. The `actions/checkout@v4` example: agent states it with confidence, it's wrong (current is v6). Every recalled fact needs a verification path. |
| K002 | M005-8pv12q planning | Pipeline design | Structure Over Instruction — telling an agent to do something is behavioral and ignorable. Structuring the pipeline so it must do the thing is architectural and reliable. ARL Principle 1. | Three iterations of M005-8pv12q design proved this: standalone templates (behavioral, ignored) → constraint labels (labeled the problem) → unknowns inventory + fact-checker (structural). |
| K003 | M005-8pv12q planning | Pipeline design | AI cannot reliably self-evaluate (ARL Principle 3). An agent classifying which of its own claims came from training data shares the same blind spots that produced them. Independent verification (fact-checker on haiku) is the structural solution. | M005-8pv12q prompts do self-classification as a useful interim. M005 adds the fact-check service layer for structural enforcement. |
| K004 | M005-8pv12q planning | Fact-check architecture | The fact-checker is an async service layer, not a blocking pipeline stage. Coordinator spawns scouts per-claim, writes annotation files, notifies orchestrator on REFUTED. Planner works with whatever is verified so far. | Blocking stage would delay planning unnecessarily. Async lets verification happen in parallel. REFUTED claims trigger revision, not gates. |
| K005 | M005-8pv12q planning | Source attribution | SAM, ARL, HOOTL are not generally known terms. They come from github.com/bitflight-devops/stateless-agent-methodology (currently private). hallucination-detector is at github.com/bitflight-devops/hallucination-detector. claude_skills is at github.com/Jamie-BitFlight/claude_skills. Always provide URLs when referencing these concepts in documentation. | User's own research and tooling. Requires proper attribution with links. |
| K006 | M006-tbhsp8 planning | Planner process design | When planning agent-executed control flow, eliminate interpretive gaps before decomposition. Every step must name the actor, observable trigger, decision condition, state-changing action, and terminal outcome. Replace vague coordination verbs with deterministic actions and durable control artifacts. | Lower-end worker models follow prose literally. Deterministic planner outputs reduce wiggle room and make orchestration reliable. |
| K007 | M007-aos64t S03 | Test execution | gsd extension tests require `npx tsx --test` instead of `node --test`. Node's `--experimental-strip-types` doesn't handle transitive `.js` imports within `.ts` files, and the gsd extension has no dist/ build output with all local imports using `.js` extensions internally. | The test runner must match the module resolution approach used by the code under test. Using tsx ensures compatibility with how the gsd extension's internal imports work. |
| K008 | M007-aos64t closeout | Runtime-proof milestone design | If a milestone claims a runtime proof, the closeout artifact should include both a deterministic fixture contract and a machine-readable validation report written by the final audit path. Console output alone is not durable enough for future milestones or debugging. | M007 only became trustworthy once S01's fixture manifest, S02's live runtime assertions, and S03's `M007-VALIDATION-REPORT.json` were combined into a repeatable proof substrate. |
| K009 | M007-aos64t S01 | Schema contract testing | Use helper functions (e.g., `assertTokenCountsEqual`, `assertInterventionsEqual`) to reduce repetition in contract tests for nested objects. 111 assertions in telemetry-contract.test.ts proved JSON round-trip fidelity for UnitMetrics. | Nested object comparisons are verbose and error-prone. Helper functions make contract tests readable and maintainable.
